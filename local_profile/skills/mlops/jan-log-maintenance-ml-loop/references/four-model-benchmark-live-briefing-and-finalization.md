# 4모델 벤치마크 실시간 브리핑 + 최종 확정 규칙

## 목적
3-way(Pipeline/Hybrid/Agent Only)와 RAG 단독 벤치가 비동기로 끝나는 동안,
중간 공유와 최종 확정 보고를 혼동하지 않도록 운영 규칙을 고정한다.

## 적용 트리거
- 사용자가 "4개 모델 전부 다시 시험" + "각각 보고서" + "통합 요약/도표"를 동시에 요청했을 때
- 장시간 벤치마크가 background process로 실행될 때

## 운영 절차
1. 실행 시작 직후
   - TODO를 `재시험 실행`, `통합요약 생성`, `도표/최종보고`로 분리
   - background session_id를 기록

2. 중간 브리핑
   - 반드시 아래 3구역으로 보고:
     - 완료: 이미 끝난 모델/산출물
     - 진행중: 실행 중 프로세스(session_id, 진행 카운터)
     - 대기: 완료 후 할 일(최종 재집계/도표 교체)
   - 통합 리포트가 선생성된 경우 `중간본(임시)`라고 명시
   - 과거 프로세스의 완료/중단 알림이 늦게 도착할 수 있으므로, 알림 본문만으로 상태를 단정하지 말고 **현재 추적 중인 session_id를 poll/log로 재확인**한 뒤 "과거 중단건" vs "최신 실행"을 분리해 브리핑한다.
   - 재시작 직후 `output_preview`/log가 비어 있을 수 있다. 이 구간은 이상으로 단정하지 말고 "시작 직후 버퍼 미형성"으로 표기하고 다음 체크포인트를 제시한다.

3. 최종 확정 게이트
   - 3-way 종료 확인 후 최신 `benchmark_summary.csv/json` 재로딩
   - RAG 최신 summary와 합쳐 4모델 통합 CSV/JSON/MD를 재생성
   - 모델별 개별 보고서 4종도 동등 조건 기준으로 재기록

4. 최종 산출물 체크
   - 표: 모델별 status/cause/runtime 핵심 지표 포함
   - 도표: 터미널 ASCII 차트(막대/순위) 포함
   - 문구: "최종본" 표기, 중간본 경로와 구분

## 권장 브리핑 템플릿
- 에이전트이름: 완료/진행중/대기 순서로 3~6줄 요약
- 에이전트이름: 파일 경로(절대경로) + 최종본/중간본 상태
- 에이전트이름: 다음 체크포인트(예: 3-way 완료 즉시 재집계)

## 피트폴
- 3-way 실행 중 산출물을 최종본처럼 보고하면 데이터 시점이 섞여 비교 신뢰도 하락
- session_id/진행카운터 미기재 시 재현성 저하
- 사용자 선호 포맷(`에이전트이름: 내용`)을 깨면 리뷰 반복 비용 증가
